# How to Generate a PyO3 Bridge for Any Python Library Using ggen

This guide walks through the full process used in `pm4py-rust` to generate a
type-safe Rust bridge for any Python library. The approach replaces hand-written
stub files with deterministic code generated from an RDF ontology.

## Prerequisites

| Tool | Where | Purpose |
|------|-------|---------|
| `ggen` | `/Users/sac/ggen/target/release/ggen` | Tera + SPARQL code generator |
| Python 3.11+ | system | Introspect the target library |
| The target Python library | pip / conda | Source of truth for API shape |
| `pyo3` + `maturin` | Cargo.toml | Rust ↔ Python FFI |

---

## Step 1 — Write the Ontology Generator Script

Create `scripts/generate_<lib>_ontology.py`. Its job is to:

1. Import the library and introspect its public API with `inspect`.
2. Classify every function into a **module**, **call style**, and **convert result**.
3. Emit a Turtle RDF file consumed by `ggen`.

### Key Classification Concepts

#### Call Style
Determines how the Rust bridge function signature is shaped.

| `callStyle` | Rust signature | When to use |
|-------------|----------------|-------------|
| `log_only` | `(py, &EventLog)` | Single structured input |
| `log_and_net` | `(py, &EventLog, &PetriNet)` | Two structured inputs |
| `log_and_model` | `(py, &EventLog)` — discovers model first | Conformance functions that need a model |
| `io_read` | `(py, path: &str)` | File read functions |
| `io_write` | `(py, &EventLog, path: &str)` | File write functions |
| `ocel_only` | `(py, ocel: &PyAny)` | Object-centric log functions |
| `prefix_and_net` | excluded from generation | Unusual arg shapes |

Replace `EventLog`/`PetriNet` with your library's input types.

#### Convert Result
Determines the Rust return type and how the Python result is extracted.

| `convertResult` | Rust return | Extraction |
|-----------------|-------------|------------|
| `unit` | `PyResult<()>` | `drop(result)` |
| `f64` | `PyResult<f64>` | `result.extract::<f64>()` |
| `fitness_dict` | `PyResult<f64>` | `result.get_item("average_trace_fitness")?.extract::<f64>()` |
| `petri_net_triple` | `PyResult<PetriNet>` | `pm4py_petri_to_rust(py, ...)` |
| `event_log` | `PyResult<EventLog>` | `pm4py_to_eventlog(py, result)` |
| `list_dict` | `PyResult<f64>` | iterate list, extract per-dict key, average |
| `dict` | `PyResult<()>` | `drop(result)` (dict with no useful scalar) |
| `string` | `PyResult<String>` | `result.extract::<String>()` |

#### The Override Pattern
`inspect.signature()` fails on `*args` functions. Add explicit override dicts
near the top of the script and apply them at the end of `classify_function()`:

```python
CALL_STYLE_OVERRIDES = {
    'my_conformance_fn':  'log_and_net',
    'my_model_check_fn':  'log_and_model',
}

CONVERT_RESULT_OVERRIDES = {
    'my_fitness_fn':      'fitness_dict',
    'my_precision_fn':    'f64',
}

def classify_function(name, fn):
    # ... auto-classification logic ...
    if name in CALL_STYLE_OVERRIDES:
        call_style = CALL_STYLE_OVERRIDES[name]
    if name in CONVERT_RESULT_OVERRIDES:
        convert_result = CONVERT_RESULT_OVERRIDES[name]
    return { ... }
```

#### The TTL Emit Loop

```python
def emit_turtle(functions):
    lines = []
    lines.append('@prefix api: <https://yourco.com/mylib-api#> .')
    lines.append('@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .')
    lines.append('@prefix xsd:  <http://www.w3.org/2001/XMLSchema#> .')
    # emit Module nodes
    for mod_name in sorted(set(f['module'] for f in functions)):
        lines.append(f'api:module-{mod_name} a api:Module ;')
        lines.append(f'    rdfs:label "{mod_name}" .')
    # emit function nodes
    for f in functions:
        uri = f'api:fn-{f["name"].replace("_", "-")}'
        lines.append(f'{uri} a api:PythonFunction ;')
        lines.append(f'    api:functionName "{f["name"]}" ;')
        lines.append(f'    api:rustName     "call_{f["name"]}" ;')
        lines.append(f'    api:inModule     api:module-{f["module"]} ;')
        lines.append(f'    api:callStyle    "{f["call_style"]}" ;')
        lines.append(f'    api:convertResult "{f["convert_result"]}" ;')
        lines.append(f'    api:paramCount   {f["param_count"]} .')
    with open('mylib-rust/ggen/ontology/mylib-api.ttl', 'w') as fh:
        fh.write('\n'.join(lines) + '\n')
```

Run the script:
```bash
python3 scripts/generate_mylib_ontology.py
```

## Step 2 — Create the ggen Project

```
mylib-rust/ggen/
├── ggen.toml
├── ontology/
│   └── mylib-api.ttl          ← generated by Step 1
├── templates/
│   ├── _helpers.rs.tera       ← static: shared conversion utilities
│   ├── mod.rs.tera            ← static: pub mod declarations
│   ├── mylib_module.rs.tera   ← dynamic: one bridge function per row
│   ├── mylib_tests.rs.tera    ← dynamic: one test per row
│   └── mylib_ocel_tests.rs.tera  ← optional: special-input tests
└── output/                    ← ggen writes here
```

### `ggen.toml` Structure

```toml
[project]
name    = "mylib-bridge-generated"
version = "0.1.0"

[ontology]
source   = "ontology/mylib-api.ttl"
base_iri = "https://yourco.com/mylib-api#"

[ontology.prefixes]
api  = "https://yourco.com/mylib-api#"
rdfs = "http://www.w3.org/2000/01/rdf-schema#"
xsd  = "http://www.w3.org/2001/XMLSchema#"

[generation]
output_dir = "output"

# One [[generation.rules]] block per output file
[[generation.rules]]
name  = "bridge-discovery"
query = { inline = """
PREFIX api: <https://yourco.com/mylib-api#>
SELECT ?functionName ?rustName ?callStyle ?convertResult ?paramCount
WHERE {
  ?fn a api:PythonFunction ;
      api:functionName  ?functionName ;
      api:rustName      ?rustName ;
      api:inModule      api:module-discovery ;
      api:callStyle     ?callStyle ;
      api:convertResult ?convertResult ;
      api:paramCount    ?paramCount .
}
ORDER BY ?functionName
""" }
template    = { file = "templates/mylib_module.rs.tera" }
output_file = "discovery.rs"
mode        = "Overwrite"
skip_empty  = false
```

Add one rule block per module (`discovery`, `conformance`, `statistics`, `io`,
etc.) and one for each test file. Add a `bridge-mod` rule that emits `mod.rs`.

## Step 3 — Write the Templates

### `_helpers.rs.tera` (static, no SPARQL loop)

Contains conversion utilities between Rust types and Python objects:
- `input_to_python(py, &RustInput) -> PyResult<PyObject>`
- `python_to_output(py, &PyAny) -> PyResult<RustOutput>`

These are called by the generated bridge functions. Write them once; ggen
copies the file unchanged.

### `mylib_module.rs.tera` (dynamic — one function per SPARQL row)

The pattern from pm4py-rust (adapt types as needed):

```tera
{% for row in sparql_results %}
{% set fn_name    = row["?rustName"]      | default(value="call_unknown") %}
{% set py_name    = row["?functionName"]  | default(value="unknown") %}
{% set call_style = row["?callStyle"]     | default(value="log_only") %}
{% set convert    = row["?convertResult"] | default(value="unit") %}

pub fn {{ fn_name }}(
    py: Python<'_>,
{%- if call_style == "log_and_net" %}
    input: &MyInput,
    model: &MyModel,
{%- elif call_style == "io_read" %}
    path: &str,
{%- elif call_style == "io_write" %}
    input: &MyInput,
    path: &str,
{%- else %}
    input: &MyInput,
{%- endif %}
) -> PyResult<{%- if convert == "petri_net_triple" %}MyModel
             {%- elif convert == "f64" or convert == "fitness_dict" %}f64
             {%- elif convert == "string" %}String
             {%- else %}()
             {%- endif %}> {
    let lib = py.import("mylib")?;
    {# ... call dispatch + result extraction ... #}
}
{% endfor %}
```

### `mylib_tests.rs.tera` (dynamic — one test per SPARQL row)

Mirror `pm4py_tests.rs.tera`: guard with `lib_available()`, provide minimal
fixtures, dispatch by `call_style`, assert by `convert`.

```bash

# 2. Run ggen

/Users/sac/ggen/target/release/ggen sync --manifest ggen.toml

```

## Step 6 — Handle Exclusions

Some functions can't be tested or bridged generically. Exclude them via the
`FILTER(?pyName NOT IN (...))` clause in each test-generating SPARQL query.

**Permanently exclude:**
- Functions that take a `Trace` or single event (not a full log)
- Functions that return networkx graphs, matplotlib figures, or other
  Python-only objects with no Rust equivalent
- Functions with `*args` that defeat signature introspection and need
  custom bridges (document them separately)
- Functions that require DataFrame columns beyond the standard event log schema

**Temporarily exclude (fix by adding overrides):**
- Functions that return inconsistent types depending on input (add a
  `CONVERT_RESULT_OVERRIDES` entry)
- Functions where pm4py auto-discovers the model first (add `log_and_model`
  to `CALL_STYLE_OVERRIDES`)

## Common Pitfalls

### "must be real number, not dict" / "KeyError"
The `convertResult` is wrong. Test the function directly in Python with the
same fixture your tests use (not a different one). Return types can differ
between a minimal fixture and a realistic log:

```bash
python3 -c "
import mylib
result = mylib.my_function(minimal_fixture)
print(type(result).__name__, result)
"
```

### Functions that always return the same type from `inspect` perspective
Some functions use `*args` and `inspect.signature()` reports no parameters.
Add them explicitly to `CALL_STYLE_OVERRIDES`.

### Generated tests fail with "object has no attribute X"
The write/read function expects a model object (e.g., BPMN graph, Petri net),
not your generic input wrapper. Add it to the NOT IN exclusion list and write
a dedicated test with the correct fixture type.

### `pm4py_to_eventlog` / round-trip helpers fail on model objects
`io_read` functions that return process models (not event logs) must have
`convertResult = 'unit'` or their own dedicated convert type. The generic
`pm4py_to_eventlog` helper assumes a DataFrame-convertible log object.

## Adapting to Non-pm4py Libraries

The approach is library-agnostic. Replace:

| pm4py concept | Your library concept |
|---------------|---------------------|
| `EventLog` | Your primary input struct |
| `PetriNet` | Your primary model struct |
| `pm4py.convert_to_dataframe()` | Your serialization method |
| `eventlog_to_pm4py()` | `input_to_python()` |
| `pm4py_to_eventlog()` | `python_to_input()` |
| `running-example.xes` | Your realistic test fixture |
| `two_activity_log()` | Your minimal test fixture |

The TTL schema (attributes `functionName`, `rustName`, `callStyle`,
`convertResult`, `inModule`, `paramCount`) is reusable as-is. Only change
the `base_iri` prefix.

## Regeneration Workflow (Day-to-Day)

When the upstream Python library releases a new version:

```bash
# 1. Update the library
pip install --upgrade mylib

# 2. Re-run the introspection script — picks up new/removed functions
python3 scripts/generate_mylib_ontology.py

# 3. Regenerate all bridge code
cd mylib-rust/ggen
/Users/sac/ggen/target/release/ggen sync --manifest ggen.toml
cp output/*.rs ../src/python/generated/
cp output/tests.rs ../../tests/mylib_official_generated_tests.rs

# 4. Compile + test — new functions appear automatically
cd ../.. && cargo test

# 5. Add any new exclusions or overrides for newly broken functions
```

New functions are picked up automatically. Removed functions produce a
compile error (the generated call site disappears) — the Rust type system
catches API drift at compile time.
